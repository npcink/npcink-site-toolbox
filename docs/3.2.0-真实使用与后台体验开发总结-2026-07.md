# Npcink Site Toolbox 3.2.0 真实使用与后台体验开发总结

> 日期：2026-07-22
>
> 状态：当前开发节点已收口，继续以真实使用反馈驱动后续修改
>
> 范围：对象存储、媒体 WebP、后台布局与反馈、收藏与运行状态、设置定位和相关验证
>
> 发布边界：本节点不构建发布包，不执行 WordPress.org/SVN 或正式发布

## 一、阶段目标与约束

本阶段不是一次发布冲刺，也不是为了增加功能数量，而是让已经存在的能力在真实 WordPress 后台中更容易找到、更容易理解、更容易验证，并修复实际使用中暴露的行为问题。

执行时采用以下边界：

- 只处理能够通过界面、真实请求或回归测试证明的问题；
- 优先完善已有功能的入口、状态、反馈和恢复能力；
- 不迁移 React/Ant Design 技术栈，不新增第三个 Vite 构建目标；
- 不引入 AI Runtime、远程队列、跨仓库工作流或新的设置页；
- 媒体处理始终保留原图和可恢复路径，不把批量能力做成不可逆的全库改写；
- 发布 ZIP、Plugin Check、最低版本和主题矩阵继续留到正式发布前执行。

这套边界的核心判断是：项目仍在 Pre-GA 阶段，可以快速修正体验，但不应以“还没发布”为理由同时扩大产品表面和长期维护成本。

## 二、本阶段形成的产品闭环

### 2.1 后台外壳与信息层级

后台外壳统一为左侧对齐、最大宽度 1200px 的工作区，去除外层重复的灰色背景和桌面端无意义边距；Header、侧栏和正文使用同一内容边界，窄屏保留安全间距。

同时处理了多处真实界面问题：

- 删除没有信息增量的副标题、面包屑和重复状态；
- 修正蓝底黑字、超宽输入组、标签与控件错位等不一致样式；
- 长表单按任务分组，常用说明保持精简，详细示例通过信息入口按需展开；
- 操作结果靠近触发按钮展示，避免用户滚动后找不到成功、失败或处理进度；
- 国内访问检测卡片、维护提示、倒计时配置和图片选择器在窄宽度下正常换行；
- 图片选择器增加分页、明确选中态和稳定的缩略图裁切，不一次性渲染整个媒体库。

视觉验收记录见 [后台界面布局视觉验收](后台界面布局视觉验收-2026-07.md)。

### 2.2 语义导航与设置状态

真实使用发现，“保存后回到默认标签”和“前往设置只打开一级页面”本质上是同一个问题：后台只保存了视图名称，没有保存用户当前任务的语义位置。

现在导航状态分为三层：

| 层级 | 作用 | 示例 |
| --- | --- | --- |
| `view` | 一级管理视图 | `site`、`maintenance` |
| `tab` | 长页面中的二级标签 | `site.media` |
| `target` | 页面内具体设置项 | `optimize-medium-upload_auto_name` |

因此：

- 在“媒体”标签保存后仍停留在“媒体”；
- 搜索、收藏和运行状态页可以直接打开具体设置项；
- 浏览器前进、后退和刷新能够恢复相同任务位置；
- 普通一级导航会主动清理过期 `tab` 和 `target`，防止旧状态污染新页面。

设置定位使用真实 `href` 作为无 JavaScript 回退，同时由 React 接管站内导航。这样既保留可访问性，也避免把深链接只实现为一次性的前端滚动。

### 2.3 收藏与运行状态

收藏按钮过去只改变星标，没有清晰的查看入口。概览页现在集中显示常用功能，并提供准确的“前往设置”路径。

“关于与帮助”增加只读的“功能与运行状态”：

- 显示插件版本、运行环境、当前加载模块和编辑器工具；
- 所有 56 个 Registry 模块必须提供用户可读中文名称；
- 普通界面不显示 `performance.oss`、`optimize.image_rename` 等内部 ID；
- 普通模块不重复显示“已启用”和默认“稳定”；
- 始终加载且没有开关的模块显示“无需开关”，也不提供误导性的设置链接；
- 内部 ID 只保留在复制的诊断报告中，方便技术支持定位。

这里形成了一条通用原则：用户界面展示任务语言，诊断报告展示实现语言，两者不能混为一层。

### 2.4 对象存储配置与连接验证

OSS 原先的问题不是缺少底层字段，而是入口、填写模型和反馈都与用户从云厂商控制台拿到的信息不一致。本阶段完成：

- 从概览和维护视图都能找到对象存储配置；
- 复杂配置放在右侧抽屉，主页面只保留启用状态和关键操作；
- Bucket 与上传目录分开，支持 `www`、`uploads/site-a` 等对象键前缀；
- Endpoint 允许直接粘贴阿里云外网地址，也接受 `cn-shanghai` 简写；
- 明确区分 Endpoint、公开访问地址/CDN 域名和上传目录；
- 提供紧凑示例、目标 URL 预览和本地文件安全回退说明；
- “测试连接”写入固定测试对象，返回对象路径、耗时和结果，但不保存设置、不改变启用状态；
- 成功或失败信息保持在操作区，重复通知被合并，页面刷新前不会立即消失。

没有在这一节点加入远程图片回源、延迟加载、本地自动删除或跨云迁移。这些能力会改变 URL 生命周期、缓存和灾难恢复边界，收益尚不足以覆盖风险。

### 2.5 新上传图片的 WebP 输出

新增“新生成图片使用 WebP”开关，复用 WordPress 原生图片编辑器的 `image_editor_output_format`：

- 只把 JPEG 的 WordPress 生成结果映射为 WebP；
- PNG 不强制有损转换；
- 不直接删除或覆盖原始上传文件；
- 当前服务器不支持 WebP 时保持 JPEG，不制造上传失败；
- 运行状态页显示当前 WordPress 图片编辑器的 WebP 支持事实。

选择 WordPress 原生图片编辑器，而不是直接操作 GD/Imagick，能继续复用核心的格式能力检测、元数据生成和失败回退，也减少对服务器扩展差异的自维护代码。

### 2.6 历史媒体批量转换

历史媒体先执行只读体检和样本估算，再允许批量转换：

- 只处理本地可读、尺寸和像素数在安全上限内的 JPEG；
- 一次任务可连续处理 10、20 或 50 张，前端自动推进；
- 服务器每个请求最多顺序处理 5 张，避免单次请求长期占用 PHP；
- 支持停止、恢复和最近批次恢复；
- 主图、缩略图、附件元数据全部成功后才切换附件；
- 原 JPEG 和转换前元数据保留，恢复失败时不删除现有文件；
- 对象存储同步失败时回退本地 URL，不把外部故障变成媒体损坏。

当前实现刻意没有做后台队列。对现阶段几十张规模，浏览器内连续处理已经消除“手工点几十批”的主要痛点，同时避免引入 Action Scheduler、Cron 队列状态、并发锁管理和失败通知等新的系统复杂度。只有真实使用证明需要离开页面后继续处理，才值得增加持久化队列。

## 三、关键技术判断

### 3.1 暂不迁移 Ant Design

Ant Design 在复杂表单、日期、抽屉和确认交互上仍有实际价值。全面迁移会同时触碰七个管理视图、响应式行为、测试和视觉基线，但不会直接解决当前用户问题。

本阶段采用更低成本的策略：

- 管理外壳、导航、状态和普通卡片使用轻量 React/CSS；
- 复杂维护视图与表单按二级标签懒加载；
- 修改到的组件逐步减少不必要的 Ant Design 包装；
- 只有 bundle 数据和真实交互证明依赖成本成为主矛盾时，再评估迁移。

框架重量是工程指标，不是单独的产品缺陷。先减少首屏加载和无效渲染，比立即替换组件库更符合当前收益比。

### 3.2 Registry、Schema 与语义目标各司其职

- Registry 决定运行模块、文件、作用域、中文名称和诊断元数据；
- Config Schema 决定设置类型、默认值、风险信息、搜索信息和敏感字段；
- `feature_id`/`target_id` 决定界面中的语义定位；
- 生成的 TypeScript 契约承担 PHP 与 React 之间的一致性检查。

不能用内部模块 ID冒充用户名称，也不能从中文文案反向推导 DOM 目标。每一种信息只保留一个事实源，测试负责阻止缺失和漂移。

### 3.3 长操作采用“小批服务端 + 连续客户端”

媒体转换采用两层批次：用户选择的是任务规模，服务端接收的是安全小批次。这个模式同时满足：

- 用户不用手工重复点击；
- PHP 请求时间和内存可控；
- 每批都有独立结果，可停止和恢复；
- 后续若升级为后台队列，转换器和单批 REST 契约仍可复用。

数据库清理、批量补 Alt 等工具也应沿用“先预览、再小批执行、结果可追踪”的路径，不应直接提供无边界的一键全库操作。

## 四、排障与验证经验

### 4.1 先复现用户路径，不从组件猜原因

本阶段多次出现“代码看起来存在，但用户找不到或跳错位置”的问题。有效排查顺序是：

1. 从用户点击的真实入口开始；
2. 记录 URL、选中的标签、目标控件和保存后的变化；
3. 再追踪导航状态、Registry/Schema 元数据和 React 组件；
4. 先写能复现旧行为的聚焦测试，再做最小修复；
5. 自动化通过后回到 Local WordPress 重新点击一次。

只检查组件是否渲染，无法证明入口正确；只检查 URL，也无法证明目标标签和设置项实际可见。

### 4.2 操作反馈必须与动作共处

成功通知如果只显示在页面顶部，长抽屉和长维护页中的用户通常看不到；短暂 toast 也不足以承载对象路径、耗时、处理数量等需要核对的信息。

本阶段统一为：

- 顶部通知只承担简短的全局状态；
- 操作区保留持续结果和错误详情；
- 进行中的按钮显示忙碌并禁止重复提交；
- 同一结果不在多个位置重复堆叠。

### 4.3 测试要覆盖契约和真实环境的差异

测试 bootstrap 可能比真实 WordPress 入口加载更多依赖；浏览器工具也可能无法读取 WordPress 7.0 的 blob 编辑 iframe。因此证据必须交叉组合：

- PHPUnit：PHP 行为、权限、输入和恢复契约；
- PHPStan：静态类型和不可达分支；
- Vitest：导航状态、组件行为、分页和批量编排；
- 构建：类型、CSS 隔离和发布用产物；
- Local：真实入口、保存/刷新、标签、目标跳转、REST 请求和控制台。

工具看不到 iframe 内部 DOM 不等于区块不存在。可以通过插入器可访问文本、保存状态、数据库 `parse_blocks()`、硬刷新和前台预览交叉验证。

### 4.4 并行门禁可能制造假失败

前端全量测试曾在与多个后端门禁同时运行时出现三个慢测试超时，但没有断言失败。将相关文件单独复测后通过，随后在无资源竞争下完整运行 42 个文件、231 项测试全部通过。

因此：

- 互不竞争的聚焦测试可以并行；
- 完整 Vitest、PHPUnit 和构建不应在资源紧张时全部并行；
- 超时应先隔离复测，不能直接归因于代码，也不能忽略；
- 最终报告必须区分第一次失败、隔离结果和最后完整结果。

推送本节点后，GitHub PHPStan 还暴露出另一种门禁漂移：Composer 已使用 2G 内存上限，但 CI 仍单独写死旧的 1G 命令，导致远端并行 worker 内存不足，而本地 `composer phpstan` 正常。最终让 CI 统一调用 Composer 脚本，并增加契约测试禁止再次复制 PHPStan 命令。质量门禁也应遵循单一入口，不能让“本地通过”和“CI 通过”维护两套参数。

覆盖率门禁也验证了同一原则：GitHub runner 上两个覆盖率 worker 最初会让多个 Ant Design/jsdom 重型测试在 5/15 秒内得不到足够 CPU；机械改成单 worker 虽在本机 179.72 秒通过，却在更慢的 runner 上串行运行 529.18 秒后仍有 8 项超时。这说明只改 worker 数是在移动资源瓶颈，不是完整修复。

最终处理保持生产代码不变：测试渲染关闭 Ant 动画；数据库清理不再真实挂载 Ant 静态确认框，而是验证本插件传入的确认合同并直接执行 `onOk`；只为 8 项确实包含分页、连续批处理或多轮弹层交互的用例设置独立 30 秒预算，其他用例仍使用默认 5 秒；覆盖率恢复为两个 worker。本机完整覆盖率最终以 42 个文件、231 项测试在 90.75 秒内通过，语句/行覆盖率 74.28%。这比提高全局超时更能保留回归灵敏度，也比单 worker 更适合 CI。

## 五、可复用检查清单

### 5.1 新增或修改后台功能

- [ ] 用户能从概览、搜索或正确分类找到入口；
- [ ] 保存、刷新、前进和后退保持合理的 `view`/`tab`/`target`；
- [ ] Registry 有用户可读 `label`，内部 ID 不直接显示；
- [ ] 简短提示与详细帮助分层，不把说明全文堆在表单中；
- [ ] 操作反馈出现在动作附近，错误可恢复；
- [ ] 320px、782px、1280px 和宽屏下无横向溢出或异常换行；
- [ ] 键盘与真实链接仍能完成主要导航；
- [ ] 聚焦测试和 Local 控制台均通过。

### 5.2 媒体批量修改

- [ ] 先只读预检候选数量、格式、体积和服务器能力；
- [ ] 服务端限制单批数量、文件体积和像素数；
- [ ] 原图、原元数据和恢复记录在切换前已经安全保存；
- [ ] 数据库、主图和缩略图全部成功后才提交状态；
- [ ] 支持停止、恢复、重复请求跳过和并发锁；
- [ ] 外部存储失败不能破坏本地可读性；
- [ ] 清理只删除本插件生成且经过上传目录边界验证的文件。

## 七、尚存风险与后续方向

### 正式发布前必须重新执行

- `docs/3.2.0-BUG排查与验证台账-2026-07.md` 中的 T9 生命周期；
- T10 WordPress 6.3/当前版本与 PHP 7.4/8.3；
- T11 当前主题与 WordPress 核心主题；
- 完整发布包构建、验证、Plugin Check 和最终 Local 安装冒烟。

### 当前明确不做

- PNG 强制有损转换；
- 自动删除原图或不可恢复覆盖；
- 持久化后台媒体队列；
- 远程图片回源、延迟加载和跨云迁移；
- 全面迁移 Ant Design；
- 为诊断页面新增独立顶级菜单；
- 为技术债数字进行全仓机械重构。

下一阶段继续以真实使用为主：发现可复现 BUG 就补回归并修复；新增功能必须先证明任务价值、边界和恢复方式。不要因为发布日期是大概时间，就把“继续开发”误解为“同时建设所有可能能力”。
