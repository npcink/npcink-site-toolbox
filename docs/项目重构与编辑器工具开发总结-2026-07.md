# Npcink Site Toolbox 2026-07 项目重构与编辑器工具开发总结

> 日期：2026-07-18
> 状态：当前有效
> 范围：3.0 复杂度收口、后台与前端工程简化、3.1 品牌统一，以及 3.2 编辑器工具开发

## 一、为什么先重构再扩展

本轮工作的起点不是“继续增加功能”，而是系统检查项目是否已经过度复杂、实际使用是否方便，以及哪些能力已经偏离站长工具箱的产品边界。

项目仍处于开发阶段，没有真实用户、历史内容或兼容负担，因此选择一次性收口而不是为错误架构继续维护兼容层。核心处理包括：

- 把模块 Registry、设置 Schema 和生成式前端契约确立为设置功能的事实源；
- 将后台收敛为语义明确的视图、搜索和按页面需要出现的二级标签；
- 删除无消费者 REST、失效功能、不可信登录验证和无实际行为的设置；
- 从正式产品和发布包移除 AI Provider Runtime，相关思路和可迁移代码只保存在非发布参考目录，供其它插件人工取用；
- 将前端收敛为一个 pnpm/Vite 工程、Admin 与 Count 两个独立产物，退役没有产品消费者的第三个 Vite 目标；
- 统一 Npcink Site Toolbox 的产品名、slug、文本域、发布包、WordPress.org 资料和品牌资产；
- 完成复杂度收口和发布边界后，才开始增加编辑器特色能力。

更细的 3.0 变更包、失败证据和验证记录见 [Pre-GA 后台重置 ADR](adr-pre-ga-admin-reset-2026-07-15.md)；公共身份边界见 [ADR-0003](decisions/0003-npcink-site-toolbox-public-identity.md)。本总结不替代这些事实记录，而是归纳跨阶段可复用的方法。

这段历史形成了一个基本判断：项目现代化不等于增加框架、页面和抽象；现代化首先是减少重复真相、明确任务入口、提供可信反馈，并让每一项新增复杂度都有用户价值依据。

## 二、编辑器工具阶段结论

3.2.0 没有把 Npcink Site Toolbox 扩展成一套新的页面构建器，而是在 WordPress 原生编辑器能力之上补齐四项面向常见内容任务的工具：三个可自由编辑的核心区块样板，以及一个读取实时站点数据的动态区块。

最终形成的产品结构如下：

| 能力 | 实现方式 | 选择理由 |
|---|---|---|
| 资源下载卡片 | 核心区块样板 | 插入后完全可编辑，不需要自定义运行时 |
| 文章结论卡片 | 核心区块样板 | 本质是内容结构，不应固化成专用区块 |
| 来源与版权说明 | 核心区块样板 | 适合复用排版，但内容应由作者直接修改 |
| 站点数据 | PHP 动态区块 | 文章、评论、分类和用户数量必须始终读取当前站点数据 |
| 概览页引导 | 紧凑的后台信息面板 | 解决功能已经存在但用户不知道去哪里使用的问题 |
| 编辑器分类 | 区块与样板使用同名品牌分类 | 在 WordPress 两套独立插入器分类中建立一致心智 |

这套方案的核心收益不是“区块数量增加”，而是用较低的维护成本验证 Npcink Site Toolbox 是否适合承载一组实用、原生、可发现的编辑器工具。

## 三、关键开发思路

### 3.1 先判断内容模型，再决定是否开发自定义区块

“想提供一个特色区块”不等于必须注册新的区块类型。首先应判断内容是否需要插件持续参与运行：

- 只是可复用的排版和文案结构：优先使用核心区块样板。
- 必须显示实时数据、执行服务端计算或统一业务规则：使用动态区块。
- WordPress 核心区块已经能直接完成：只写使用说明，不再包装一层。

三个内容卡片全部由核心区块组成。用户插入后得到的是普通 WordPress 内容，即使以后停用插件，已经写入文章的核心区块仍然可以继续编辑和显示。这比为每张卡片维护自定义属性、保存标记和废弃版本更符合工具箱的轻量定位。

“站点数据”不同。统计值会随站点变化，若保存为静态内容就会过期，因此使用 `block.json` 注册元数据、编辑器脚本提供配置和预览、PHP `render_callback` 输出实时结果。

### 3.2 实时能力只保留一个事实源

项目原有“站点统计”小工具已经需要文章、评论、分类和用户数量。新增动态区块时，没有复制四套查询，而是把统计读取和 HTML 列表渲染集中到 `MaBox_Site_Stats`：

- `get_stats()` 在一次请求内缓存统计快照。
- `render_items()` 同时服务动态区块和旧小工具。
- `render_block()` 只处理区块属性、标题和空选择状态。

这样避免了小工具与区块在统计口径、转义和性能行为上逐渐分叉。新增消费者时，应先判断能否复用现有领域服务，而不是从界面层重新查询数据。

### 3.3 不为一个小区块增加第三个 Vite 构建目标

后台 Admin 和 Count 已经是两个独立产物。动态区块的编辑器逻辑只有少量原生 WordPress 依赖，因此直接发布可读的 `blocks/site-stats/index.js` 和 `index.asset.php`，没有再创建 `vite/blocks`、新锁文件或第三套构建命令。

这一选择减少了：

- Node 依赖和构建配置漂移；
- CI、发布 ZIP 和本地开发命令的维护面；
- 为少量代码支付的长期工具链成本。

只有当区块数量、共享组件或 TypeScript 需求已经明显超过可读源码的维护能力时，才应重新评估独立构建产物；不能因为“现代区块通常要构建”就提前引入工程。

### 3.4 功能完成不等于用户能够发现

样板和动态区块最初只存在于文章或页面编辑器中，而插件的主要管理入口在后台设置页。用户可能知道插件新增了能力，却不知道在哪里使用。

因此概览页增加一块全宽“编辑器工具”引导面板：

- 说明当前有三个样板和一个动态区块；
- 明确这些工具只在文章或页面编辑器中使用；
- 提供“新建文章”和“新建页面”的直接入口；
- 不增加新的顶级导航、管理页或设置项。

实现中复用了现有概览面板和 WordPress 原生按钮。曾考虑过更强的图标和专用按钮系统，但它们不能提高任务完成率，最终被简化。后台工具界面的现代化重点应是信息层级、清晰动作和状态反馈，而不是增加装饰系统。

### 3.5 区块分类与样板分类是两套系统

WordPress 的“区块”和“区块样板”位于插入器的不同标签，并使用不同注册机制：

- 区块分类通过 `block_categories_all` 提供，并由 `block.json.category` 引用。
- 样板分类通过 `register_block_pattern_category()` 注册，并由样板定义中的 `categories` 引用。

因此无法创建一个同时物理容纳区块和样板的分类。正确做法是在两套系统中使用相同的 slug 和显示名 `Npcink Site Toolbox`：

- “区块”标签的同名分类中显示“站点数据”。
- “区块样板”标签的同名分类中显示三个样板。

样板仍保留 `text`、`call-to-action` 等核心分类作为第二入口，兼顾品牌聚合和按用途发现。区块分类注册保持幂等，避免同一 slug 被重复追加。

## 六、验证经验

### 6.1 自动化验证要覆盖不同边界

单一测试层无法证明编辑器能力真正可用。本阶段采用分层门禁：

1. PHP 契约测试检查 Hook、分类、样板文件、元数据注册、动态渲染、转义和空状态。
2. JavaScript 契约测试在无浏览器环境执行编辑器脚本，检查注册名、动态 `save() === null`、控件数量和属性更新。
3. 前端质量门禁验证 Admin/Count 类型、测试、构建和 CSS 隔离。
4. 发布包验证确保 `blocks/site-stats` 的源码、依赖声明和样式实际进入 ZIP。
5. Local WordPress 实机验收检查插入器分类、样板预览、区块渲染和控制台。

分类收口后的当前验证基线为：

- 聚焦 PHPUnit：8 项测试、74 个断言通过。
- 完整 PHPUnit：479 项测试、5523 个断言通过。
- PHPStan：无错误。
- 编辑器脚本契约：通过。
- Local 文章编辑器：“Npcink Site Toolbox”区块分类包含“站点数据”，同名样板分类包含 3 个样板。
- Local 浏览器控制台：无新增错误或警告。

### 6.2 实机验证不能被单元测试替代

注册表、测试 bootstrap 和真实 WordPress 入口的加载路径可能不同。涉及 Hook、编辑器脚本、分类和资源加载时，自动化通过后仍应在真实 WordPress 编辑器检查一次。

烟测应尽量避免污染内容：可以打开已有测试文章，只检查分类和已存在区块的渲染；若无需验证保存路径，不要创建自动草稿或点击保存。需要验证插入和持久化时，再使用专门的测试文章，并在验收记录中说明清理结果。

真实后台问题的另一条经验是：测试 bootstrap 可能主动加载真实入口没有加载的依赖，从而掩盖运行时故障。REST 和 Dashboard 的完整案例见 [后台 REST 与 Dashboard 排障经验](后台REST与Dashboard排障经验-2026-07-15.md)。

## 七、复杂度与协作控制原则

后续特色区块开发继续遵循以下顺序：

1. **先证明用户任务。** 能明确说出区块帮助用户更快完成什么，而不是只描述一种视觉效果。
2. **优先核心能力。** 核心区块能完成就不包装；样板能完成就不开发自定义区块。
3. **动态行为必须有理由。** 只有实时数据、服务端规则或无法保存为静态内容的行为才进入动态区块。
4. **复用领域服务。** 不允许界面消费者各自维护同一份查询和业务口径。
5. **一个工作包只做一个区块。** 实现、测试、实机验收和发布包验证完成后再扩展下一个。
6. **不提前建设管理系统。** 目前不需要区块开关页、模板市场、独立区块管理导航或远程模板服务。
7. **构建复杂度随真实规模增长。** 在共享 TypeScript 组件和多个区块确有需求前，继续使用可读、无构建源码。

可以用下面的判断表决定实现方式：

| 问题 | 是 | 否 |
|---|---|---|
| WordPress 核心区块已经直接满足需求？ | 写使用说明或样板 | 继续判断 |
| 只是复用布局和内容结构？ | 核心区块样板 | 继续判断 |
| 必须读取实时数据或执行服务端规则？ | 动态区块 | 谨慎评估静态自定义区块 |
| 会引入新的设置页、服务或构建目标？ | 先证明收益高于长期成本 | 保持当前轻量实现 |

### 7.1 多代理并行开发的边界

此前多个工作包可以并行推进，但前提是先按模块和文件区域划分责任：

- 每个代理只负责一个明确模块，并提前声明预计修改的文件；
- 公共入口、版本文件和共享契约由统筹者集中集成，避免多人同时覆盖；
- 子任务分别运行聚焦测试，统筹者对合并后的完整差异运行全量门禁；
- 暂存和提交只由一个统筹者执行，提交前核对精确文件清单；
- 并行的目标是缩短等待时间，而不是同时增加产品表面。

这种方式把代理当作受边界约束的并行执行者，而不是让多个实现者各自重构全局。文件区域划分、共享契约集中管理和最终统一验收缺一不可。

## 八、下一阶段建议

当前四项工具已经形成完整的最小闭环，下一阶段不应立即批量增加区块。建议先观察和人工使用现有工具，再选择一个高频、边界清楚、WordPress 核心无法直接满足的场景进行单区块试验。

新候选进入开发前至少应满足：

- 有明确使用场景和目标用户；
- 无需把 Core、Cloud、AI Runtime 或远程工作流职责带入本插件；
- 能复用现有 WordPress API 或项目服务；
- 不依赖新的全局设置表面；
- 可以用自动化契约和 Local 编辑器完成闭环验证。

若候选只能带来“看起来更丰富”，却需要新的构建工程、管理入口和长期兼容代码，应暂缓。Npcink Site Toolbox 的特色应来自高频任务完成效率，而不是自定义区块数量。
