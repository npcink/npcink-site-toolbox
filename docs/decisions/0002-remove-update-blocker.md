# ADR-0002: 从 WordPress.org 插件移除更新阻断器

## 状态

已接受

## 日期

2026-07-16

## 背景

`optimize.ban_update` 会停止 WordPress 核心、插件和主题的更新检查，清除相关定时任务，并隐藏更新提示。此行为既会阻断安全更新，也不符合 WordPress.org 对目录插件不得干预官方更新机制的要求。

3.0.1 尚未提交 WordPress.org，当前没有用户，也没有需要保留的兼容或历史配置负担。此时可以直接收缩产品边界，不需要保留旧类、配置迁移、隐藏开关或兼容 shim。

## 决策

- 从插件中删除 `optimize.ban_update` 的实现文件和 `MaBox_Ban_Update` 类。
- 同步删除 Autoload、Registry、Tier、Config Schema、默认值、搜索索引、前端契约和发布产物中的对应入口。
- 当前 Registry 从 57 个模块调整为 56 个，功能搜索索引从 33 条调整为 32 条。
- 不在本插件中保留停用代码、隐藏开关或兼容路径。

## 理由

站点核心、插件和主题更新是安全修复的主要交付渠道。一个通用 WordPress.org 工具箱不应让站点所有更新提示和检查静默失效，也不应让管理员误以为站点仍会获得安全补丁。

在没有用户和兼容负担的阶段直接删除，比保留不可公开发布的高风险能力更简单，也能让 Registry、Schema、搜索与前端保持一个真实边界。

## 备选方案

### 保留为高风险开关

拒绝。风险标签不能消除对更新机制的干预，也不能满足 WordPress.org 的发布边界。

### 只隐藏前端入口，保留后端代码

拒绝。休眠代码仍可能被配置或未来改动重新激活，并继续扩大安全和审查面。

### 改为仅禁止自动更新

拒绝。本插件没有足够产品理由拥有更新策略控制权；站点更新策略应由专门运维工具承担。

## 影响

- 3.0.1 不再提供禁用 WordPress、插件或主题更新的设置。
- 不迁移或读取旧的 `optimize.site.renew`，因为当前没有用户数据需要兼容。
- 发布包、后台契约和搜索结果不再包含 `optimize-site-renew`。
- WordPress 的原生更新检查、提示和安全更新渠道保持可用。

## 回滚

不把更新阻断器恢复到 Magick Toolbox。若未来在明确受控、非 WordPress.org 的环境中确有需要，应将实现恢复为独立插件，由使用方单独安装和维护，并明确告知安全风险、更新恢复步骤和责任边界。
