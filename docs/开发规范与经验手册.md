## 一、代码组织规范

### 1.1 文件加载顺序

```
Magick_Mixtrue::load_dependencies() 必须按以下顺序加载：

1. includes/class-magick-helpers.php    ← 公共工具类（最先加载）
2. includes/class-magick-mixtrue-tool.php
3. admin/class-magick-mixtrue-admin.php
4. public/class-magick-mixtrue-public.php
```

**原因**：Helpers 类被所有模块依赖，必须最先加载。Admin 类依赖 Helpers 和 Tool，必须后加载。

### 1.2 模块文件命名规范

| 层级 | 命名规则 | 示例 |
|---|---|---|
| 类名 | `Npcink_` 或 `MaBox_` 前缀 + 功能描述 | `Npcink_Hide_Top_Toolbar` |
| 文件名 | 小写下划线，与类名对应 | `hide_top_toolbar.php` |
| 方法名 | 小写下划线 | `enqueue_assets()` |
| 常量 | 全大写下划线 | `MAGICK_MIXTURE_NAME` |

### 1.3 类声明保护

所有类必须包裹在 `class_exists()` 检查中：

```php
if (!class_exists('Npcink_Module_Name')) {
    class Npcink_Module_Name {
        // ...
    }
}
```

## 二、资源加载规范

### 2.1 禁止直接输出 `<script>` / `<style>` 标签

**❌ 禁止**：
```php
public static function run() {
?>
    <script>/* JS */</script>
    <style>/* CSS */</style>
<?php
}
```

**✅ 正确**：
```php
public static function run() {
    add_action('wp_enqueue_scripts', array(__CLASS__, 'enqueue_assets'));
}

public static function enqueue_assets() {
    wp_enqueue_script(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION, true);
    $js = "/* JS */";
    wp_add_inline_script(MAGICK_MIXTURE_NAME . '_handle', $js);

    wp_enqueue_style(MAGICK_MIXTURE_NAME . '_handle', '', array(), MAGICK_MIXTURE_VERSION);
    $css = "/* CSS */";
    wp_add_inline_style(MAGICK_MIXTURE_NAME . '_handle', $css);
}
```

### 2.2 资源句柄命名规则

```
MAGICK_MIXTURE_NAME . '_' . 功能标识

示例：
magick-optimize_ban_copy          ← 禁止复制
magick-optimize_dynamic_title     ← 动态标题
magick-optimize_OwO-inline        ← 评论表情
magick-optimize_runcode           ← 运行代码
```

**唯一性保证**：同一句柄名在整个插件中只能注册一次。

### 2.3 动态值安全转义

| 上下文 | 转义函数 | 示例 |
|---|---|---|
| JS 字符串 | `esc_js()` | `$value = esc_js($option);` |
| HTML 属性 | `esc_attr()` | `content="<?php echo esc_attr($value) ?>"` |
| HTML 内容 | `wp_kses_post()` | `$content = wp_kses_post($html);` |
| URL | `esc_url()` | `href="<?php echo esc_url($url) ?>"` |

### 2.4 条件性资源加载

```php
public static function enqueue_assets() {
    if (!is_user_logged_in()) {
        wp_enqueue_style(/* ... */);
        wp_add_inline_style(/* ... */);
    }
}
```

**注意**：条件判断必须在 `enqueue_assets()` 内部，而非在 `run()` 中决定是否添加 hook。

## 三、按需加载规范

### 3.1 路由表生成规则

`build_active_modules_map()` 中每个模块的映射必须满足：

1. **配置键名**：必须与实际 UI 组件中的 `dataIndex` / `name` 一致
2. **文件路径**：相对于 `admin/partials/` 的路径
3. **Handler 注册**：`activate_module()` 中必须有对应的闭包函数

**新增模块三步走**：
```
1. 在 build_active_modules_map() 中添加配置 → 路径映射
2. 在 activate_module() 的 $handlers 数组中添加路径 → 激活逻辑
3. 在路由表中确认路径与实际文件匹配
```

### 3.2 降级回退一致性

`load_legacy()` 中的模块加载条件必须与 `build_active_modules_map()` 中的条件一致：

```php
// build_active_modules_map() 中：
if (wp_is_mobile() && !empty($config['h5'])) {
    $active[] = 'h5.php';
}

// load_legacy() 中必须保持一致：
if (wp_is_mobile()) {
    require_once /* ... */ 'partials/h5.php';
    MaBox_H5::run();
}
```

### 3.3 配置写入点同步

**所有修改配置的入口点都必须重建路由表**：

| 入口点 | 方法 | 路由表重建 |
|---|---|---|
| AJAX 保存 | `save_option_wmt_callback()` | ✅ 已实现 |
| 导入设置 | `import_settings_callback()` | ✅ 已实现 |
| REST API 保存 | `rest_save_settings()` | ✅ 已实现 |
| 插件激活 | `register_activation_hook()` | ✅ 初始化为空数组 |

**新增配置写入点时**：必须在保存成功后调用 `build_active_modules_map()` 并更新选项。

## 五、React/Vite 开发规范

### 5.1 组件懒加载

所有 Tab 组件必须使用 `React.lazy()` + `Suspense`：

```tsx
const Dashboard = lazy(() => import("@/components/dashboard/index"));
const TabFallback = <div style={{ padding: "48px" }}><Spin size="large" /></div>;

{
  key: "0",
  label: "仪表盘",
  children: <Suspense fallback={TabFallback}><Dashboard /></Suspense>,
}
```

### 5.2 Vite 分包配置

每个 Tab 对应一个独立 chunk：

```typescript
manualChunks: {
  'tab-dashboard': ['./src/components/dashboard/index'],
  'tab-page': ['./src/components/page/index'],
  'tab-optimize': ['./src/components/optimize/index'],
  // ... 其他 tab
  'vendor': ['react', 'react-dom', 'antd'],
},
```

**新增 Tab 时**：
1. 在 `manualChunks` 中添加 `'tab-xxx': ['./src/components/xxx/index']`
2. 在 `tab.tsx` 中添加 lazy import
3. 在 `items` 数组中添加 Tab 配置

### 5.3 DataContext 使用

```tsx
const { optionData, updateOption } = useContext(DataContext);

// 读取
const siteConfig = optionData.optimize?.site;

// 更新（仅支持两级嵌套）
updateOption("optimize", "site", newSiteConfig);
```

**限制**：`updateOption` 仅支持 `optionData[father][son]` 两级更新。深层嵌套需自行处理。

### 5.4 预设应用模式

```tsx
const applyPreset = async (presetId: string) => {
  const preset = presets.find(p => p.id === presetId);
  if (!preset) return;

  // 1. 深合并
  const merged = deepMerge(JSON.parse(JSON.stringify(optionData)), preset.config);

  // 2. 更新 UI 状态
  Object.keys(preset.config).forEach((father) => {
    Object.keys(preset.config[father]).forEach((son) => {
      updateOption(father, son, merged[father][son]);
    });
  });

  // 3. 持久化
  await saceOption(merged);
  message.success(`已应用预设：${preset.name}`);
};
```

## 六、常见陷阱与规避

### 6.1 方法名冲突

**问题**：同一类中出现两个同名方法 → PHP Fatal Error

**规避**：
- 替换方法时，先删除旧方法再添加新方法
- 提交前运行 `grep -n "function methodName()" file.php` 检查重复
- 每次修改后运行 `php -l file.php`

### 6.2 wp_add_inline_script 时序

**问题**：在 `the_content` 过滤器中调用 `wp_enqueue_script`

**原因**：`wp_enqueue_scripts` action 在 `the_content` 之前已执行

**规避**：
- 条件性脚本直接在过滤器中调用 `wp_enqueue_script` + `wp_add_inline_script`
- WordPress 允许在 `wp_footer` 之前随时入队 footer 脚本

### 6.3 wp_add_inline_style 时序

**问题**：在 `comment_form_defaults` 过滤器中调用 `wp_add_inline_style`

**原因**：样式已在 `wp_head` 中输出

**规避**：
- 对于必须在 `wp_head` 之后添加的样式，保留 inline `<style>` 输出
- 或在 `wp_enqueue_scripts` 阶段预先注册样式和 inline 数据

### 6.4 路由表键名拼写

**问题**：`'Magick_ToolBox_Acttive_Modules'` 拼错为双 t

**规避**：
- 所有选项键名必须定义为常量
- 禁止在代码中硬编码字符串

```
1. 确认模块是否在路由表中
2. 修改 PHP 逻辑
3. 修改对应的 React 组件
```

### 9.3 迁移遗留 inline 资源

**优先级排序**：
1. 高频页面加载的资源（前台 JS/CSS）
2. 体积大的资源（> 5KB）
3. 低频资源（admin 编辑器内联 JS）

1. 读取文件，识别 inline `<script>` / `<style>`
2. 创建 `enqueue_assets()` 方法
3. 将 JS/CSS 内容提取为 PHP 字符串
4. 使用 `wp_add_inline_script/style` 注入
5. 删除旧的 echo 方法
6. 运行 `php -l` 验证

## 十、经验教训

1. **替换方法时先删后加** — 永远不要在同一类中保留两个同名方法
2. **所有配置写入点都要同步** — 路由表重建必须覆盖所有入口
3. **降级路径与正常路径等价** — 回退模式行为必须一致
4. **常量优于字符串** — 选项名、配置键应定义为常量
5. **分批迁移优于一次性** — 大规模迁移应分步进行
6. **构建产物不入库** — dist/ 目录应始终 gitignore
7. **先验证再提交** — 每个 PHP 文件修改后必须运行 `php -l`
8. **预设应用要考虑状态同步** — 深合并后必须同时更新 UI 和持久化数据
9. **wp_add_inline_* 有时序限制** — 样式在 wp_head 后无法注入，脚本在 wp_footer 前均可
10. **条件性资源判断在 enqueue 内部** — 不在 run() 中决定是否添加 hook
11. **新增功能必须同步 5 个位置** — PHP 文件、build_active_modules_map、activate_module handlers、interface.tsx、defaultVar.tsx、React UI 组件，缺一不可

## 十一、批量新增功能经验总结（v2.0.85 — 12 项功能一次性落地）

> 本次开发一次性落实了 12 项功能，涵盖 P0/P1/P2 三个优先级。以下是实战中总结的经验和踩过的坑。

### 11.1 新增功能标准流程（5+2 步法）

每个新功能的落地必须完成以下 7 个步骤，**缺一不可**：

```
步骤 1: PHP 文件 — admin/partials/[category]/[feature].php
        静态类 + class_exists 守卫 + run($config) 方法

步骤 2: 路由表映射 — admin/class-magick-mixtrue-admin.php
        build_active_modules_map() 中添加 配置键 → 文件路径 映射

步骤 3: 模块激活器 — admin/class-magick-mixtrue-admin.php
        activate_module() 的 $handlers 中添加 路径 → 闭包函数 映射

步骤 4: TypeScript 类型 — vite/admin/src/tool/interface.tsx
        在对应类型中添加新字段 + 行尾注释

步骤 5: 默认值 — vite/admin/src/tool/defaultVar.tsx
        在对应默认对象中添加字段 + 默认值 + 行尾注释

步骤 6: React UI — vite/admin/src/components/[tab]/[component].tsx
        添加 Ant Design 表单控件，绑定 formData.xxx

步骤 7: 短代码注册（如适用）— shortcode/compose/index.php
        在 runs() 方法中添加 require_once + ::run() + $option_list
```

**经验**：步骤 2 和 3 最容易遗漏。新增功能后如果后台开关不生效，90% 是因为路由表或 handlers 没注册。

### 11.2 不同类型功能的最佳实践

#### 短代码类功能（Bilibili、打赏、公众号解锁）

```php
// 标准模板
class MaBox_ShortCode_Xxx {
    public static function run() {
        add_shortcode('mabox_xxx', array(__CLASS__, 'render'));
        // 条件性加载资源：只在包含短代码的文章中加载
        add_action('wp_enqueue_scripts', function () {
            global $post;
            if (isset($post) && has_shortcode($post->post_content, 'mabox_xxx')) {
                self::load_assets();
            }
        });
    }
}
```

**关键点**：
- 使用 `has_shortcode()` 做条件加载，避免全站加载不必要的 CSS/JS
- 短代码命名统一 `mabox_` 前缀，如 `mabox_bilibili`、`mabox_reward`、`mabox_wx_unlock`
- 渲染时使用 `shortcode_atts()` 提供默认参数，支持用户覆盖

#### 评论过滤类功能（敏感词、百度审核）

```php
// 挂载 preprocess_comment 过滤器
add_filter('preprocess_comment', array(__CLASS__, 'check_comment'));

// 拦截时使用 wp_die() + back_link
wp_die(
    esc_html__('您的评论包含敏感词，请修改后重新提交。'),
    esc_html__('评论被拦截'),
    array('back_link' => true)
);
```

**关键点**：
- `preprocess_comment` 是评论提交的最早钩子，适合做内容审核
- 百度审核 API 的 access_token 必须用 `set_transient()` 缓存，避免每次请求都重新获取
- 本地敏感词库用 `mb_stripos()` 做不区分大小写的匹配，比 `stristr()` 更安全

#### 前端效果类功能（阅读进度条）

```php
// 仅在文章页展示
public static function render_bar() {
    if (!is_single()) {
        return;
    }
    // ...
}
```

**关键点**：
- 使用 `is_single()` 限制作用域，避免在首页、分类页等不必要的渲染
- CSS 使用 `position: fixed` + `z-index: 999999` 确保在最上层
- JS 使用 `{ passive: true }` 监听 scroll 事件，避免性能问题

#### 安全防护类功能（进阶防刷）

```php
// 使用 Transient 做 IP 级别的请求计数
$transient_key = 'mabox_anti_crawler_' . md5($ip);
$data = get_transient($transient_key);
```

**关键点**：
- IP 获取优先使用 `HTTP_X_FORWARDED_FOR`（反代场景），回退到 `REMOTE_ADDR`
- 腾讯防水墙验证结果必须服务端二次校验（`ssl.captcha.qq.com/ticket/verify`）
- 验证通过后设置 Cookie 标记，避免用户每次触发都要验证

### 11.3 微信生态功能开发经验

#### 微信内检测与优化

```php
// 两种模式：alert（弹窗）vs optimize（引导条+小程序入口）
$mode = MaBox_Admin::get_config(self::$option, 'ban_open_weixing_mode', 'alert');

// 检测微信 UA
if (ua.match(/MicroMessenger/i) == "micromessenger") {
    document.body.classList.add('mabox-wechat-mode');
}
```

**经验**：
- 旧版 `ban_open_weixing.php` 仅做 `alert()` 提示，用户体验差
- 新增 `optimize` 模式：隐藏 WP admin bar，显示引导条，支持小程序入口
- 小程序链接使用 `weixin://dl/business/...` 格式

#### 公众号验证码解锁

```php
// 解锁状态通过 Cookie 持久化（1 年有效期）
setcookie('mabox_wx_unlocked', '1', time() + YEAR_IN_SECONDS, COOKIEPATH, COOKIE_DOMAIN, is_ssl(), true);

// 验证码列表存储在配置中，每行一个
$codes = explode("\n", trim($config['unlock_codes']));
```

**经验**：
- 验证码验证使用 AJAX（`wp_ajax_nopriv_` 钩子），避免页面刷新
- 内容区域使用 `filter: blur(8px)` 做视觉遮挡，比 `display: none` 更有"解锁"感
- 验证码存储在配置项中，适合小站长手动管理；大站应接入数据库

### 11.4 CDN 替换功能经验

```php
// Gravatar 替换需要同时处理两个过滤器
add_filter('get_avatar', array(__CLASS__, 'replace_gravatar'));
add_filter('get_avatar_url', array(__CLASS__, 'replace_gravatar_url'));

// Google Fonts 替换
add_filter('style_loader_src', array(__CLASS__, 'replace_google_fonts'));
```

**经验**：
- Gravatar 有多个 URL 来源（www、0、1、2、secure、cn），必须全部替换
- Google Fonts 的字体文件在 `fonts.gstatic.com`，也需要替换为 `gstatic.loli.net`
- 自定义 CDN 替换支持 `原地址 => 新地址` 格式，同时作用于 `style_loader_src` 和 `script_loader_src`
- 推荐镜像：`fonts.loli.net`、`gravatar.loli.net`、`ajax.loli.net`（中科大 Loli CDN）

### 11.5 性能优化经验

#### 配置内存缓存

```php
// 问题：get_seting() 每次调用都执行 get_option() + 反序列化
// 解决：使用静态属性做请求级缓存
private static $config_cache = null;

public static function get_seting($option) {
    if (self::$config_cache === null) {
        self::$config_cache = get_option(MAGICK_MIXTURE_OPTION);
    }
    return self::get_config(self::$config_cache, $option);
}
```

**注意**：
- 这是**请求级**缓存，不是跨请求缓存。每次页面加载只反序列化一次
- 保存配置后缓存自动失效（下次请求重新读取）
- 如需手动清除，提供 `clear_config_cache()` 方法

#### 微信 Token 缓存

```php
// 微信 access_token 有效期 2 小时，必须缓存
$cached = get_transient('mabox_wx_token');
if ($cached) {
    return $cached;
}
// 获取新 token 后缓存 7000 秒（留 200 秒余量）
set_transient('mabox_wx_token', $token, 7000);
```

#### 百度审核 Token 缓存

```php
// 百度 access_token 有效期 30 天
$expires = isset($body['expires_in']) ? intval($body['expires_in']) - 300 : 2592000;
set_transient('mabox_baidu_moderation_token', $body['access_token'], $expires);
```

### 11.6 REST API 安全修复

```php
register_rest_route('wx_xcx/v1', 'qy', array(
```

## 十二、响应式适配与安全增强实战经验（v2.0.86）

> 本次开发依次落实了 5 项改进：卸载清理补全、百度审核降级、敏感词拼音变体、阅读进度条验证、设置页响应式适配。

### 12.1 卸载清理规范

**问题**：`uninstall.php` 清理了 `Magick_ToolBox_Option`，但遗漏了 `MAGICK_TOOLBOX_ACTIVE_MODULES`。

**规范**：每新增一个 `wp_options` 键，必须在 `uninstall.php` 中同步添加 `delete_option()`。

```php
// uninstall.php 检查清单
delete_option('Magick_ToolBox_Option');
delete_option(MAGICK_TOOLBOX_ACTIVE_MODULES);  // 使用常量，禁止硬编码
// 新增选项时，在此追加 delete_option()
```

**经验**：在 `magick-tool-box.php` 的激活钩子中定义的所有 `add_option()` / `update_option()`，都必须在 `uninstall.php` 中有对应的 `delete_option()`。

### 12.2 第三方 API 降级策略

**场景**：百度文本审核 API 可能因网络、Key 过期、限流等原因不可用。

**策略**：API 失败时不应直接放行评论，而应降级到本地敏感词过滤。

```php
// 降级模式实现
private static function fallback_filter($commentdata) {
    // 复用本地敏感词逻辑
    $raw_words = MaBox_Admin::get_config(self::$option, 'sensitive_words', '');
    if (empty($raw_words)) {
        return $commentdata;  // 无本地词库则放行
    }
    // ... 执行敏感词过滤
}

// 在 API 调用失败时调用
if (!$token) {
    error_log('[MaBox] Baidu moderation token acquisition failed.');
    return self::fallback_filter($commentdata);
}

if ($result === 'unknown') {
    error_log('[MaBox] Baidu moderation API returned unknown result.');
    return self::fallback_filter($commentdata);
}
```

**规范**：
- 所有第三方 API 调用必须有降级路径
- 降级失败必须记录 `error_log()`，方便站长排查
- 降级路径应复用已有的本地逻辑，避免重复代码

### 12.3 敏感词双向匹配机制

**问题**：用户通过拼音、缩写规避敏感词检测（如"微信"→"vx"、"V信"）。

**方案**：建立拼音变体映射表，支持双向匹配。

```php
private static $pinyin_variants = array(
    '微信' => array('vx', 'v信', '威信', '薇信', 'vxin', 'weixin'),
    '支付宝' => array('zfb', '支福宝', '芝麻信用'),
    // ...
);

// 正向匹配：敏感词 → 拼音变体
if (isset(self::$pinyin_variants[$word])) {
    $expanded = array_merge($expanded, self::$pinyin_variants[$word]);
}

// 反向匹配：拼音变体 → 敏感词
foreach (self::$pinyin_variants as $original => $variants) {
    if (in_array(strtolower($word), array_map('strtolower', $variants))) {
        $expanded[] = $original;
    }
}
```

**经验**：
- 映射表应覆盖国内主流平台/应用的常见变体
- 使用 `strtolower()` + `in_array()` 做不区分大小写的反向匹配
- 最终用 `array_unique()` 去重，避免重复替换

### 12.4 前端效果移动端兼容性验证

**阅读进度条验证结论**：

| 检查项 | 结果 | 说明 |

本次适配涉及四个文件，按影响范围从大到小：

#### 12.5.1 表单布局响应式（tool.tsx）

#### 12.5.3 导航标签响应式（tab.tsx）

```tsx
// 监听窗口宽度
const [isMobile, setIsMobile] = useState(window.innerWidth < 768);
useEffect(() => {
    const handleResize = () => setIsMobile(window.innerWidth < 768);
    window.addEventListener("resize", handleResize);
    return () => window.removeEventListener("resize", handleResize);
}, []);

// 动态切换标签位置
<Tabs tabPosition={isMobile ? "top" : "left"} items={items} />
```

**效果**：移动端标签在顶部横向排列，桌面端在左侧纵向排列。

**经验**：
- 使用 `useEffect` + `resize` 事件监听，注意清理（`removeEventListener`）
- 阈值 768px 与 CSS `@media (max-width: 768px)` 保持一致
- 不需要引入第三方库（如 `use-media`），原生实现足够简洁

## 十三、模块注册表规范（2.1.0+）

### 13.1 注册表字段定义

所有模块必须在 `admin/modules/registry.php` 中注册：

```php
'module.id' => array(
    'class'             => 'Npcink_Module_Class',  // 必须：类名
    'file'              => 'path/to/file.php',      // 必须：相对于 partials/ 的路径
    'option_key'        => 'path.to.switch',        // 必须：开关的选项路径（点分格式）
    'category'          => 'optimize',              // 必须：模块分类
    'scope'             => 'frontend',              // 必须：frontend / admin / both
    'config_path'       => 'path.to.config',        // 可选：传递给 run() 的配置路径
    'always_load'       => true,                    // 可选：始终加载（短代码/模板）
    'mobile_only'       => true,                    // 可选：仅移动端加载
    'theme_requirement' => 'B2',                    // 可选：需要特定主题
    'risk_tags'         => array('推荐', 'SEO'),    // 可选：风险标签
),
```

### 13.2 新增功能流程

```
1. 在 registry.php 添加模块记录
2. 确保 class 和 file 字段指向正确的类名和文件路径
3. 确保 option_key 与 React 端表单字段对应
4. 如需要传递完整配置给 run()，添加 config_path 字段
5. 添加 risk_tags 标签
6. 在 feature-search.tsx 中添加搜索索引
```

### 13.3 路径计算规范

**禁止多层 dirname() 嵌套**：

```php
// ❌ 错误：难以维护，容易出错
$build_css = plugin_dir_url(dirname(dirname(dirname(__DIR__)))) . 'vite/count/dist/index.css';

// ✅ 正确：使用 plugin_dir_path(__FILE__) 配合相对路径
$build_css = plugin_dir_url(__FILE__) . '../../../../../vite/count/dist/index.css';

// ✅ 最佳：定义常量或使用 helper
$plugin_url = plugin_dir_url(dirname(dirname(__DIR__)));
$build_css = $plugin_url . 'vite/count/dist/index.css';
```
