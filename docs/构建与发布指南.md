# 构建与发布指南

## 本地开发

### 前置要求

- Node.js >= 18.12.0
- pnpm（版本由 `vite/package.json` 的 `packageManager` 固定）
- PHP >= 7.4
- Composer

### 安装依赖

```bash
# PHP 依赖
composer install

# 前端依赖
corepack enable
cd vite
pnpm install --frozen-lockfile
```

### 开发模式

```bash
cd vite

# 启动管理面板开发服务器
pnpm run dev:admin

# 启动计数器开发服务器
pnpm run dev:count
```

`vite/` 只有一份根 `package.json`、锁文件和质量工具链。`admin/` 与 `count/` 保留各自的 Vite 配置和独立输出目录，但不是独立 package。

### 构建前端资源

```bash
cd vite

# 构建全部目标
pnpm run build

# 或单独构建一个目标
pnpm run build:admin
pnpm run build:count

# 共享质量门禁
pnpm run typecheck
pnpm run lint
pnpm run test
```

## 发布流程

### 唯一本地发布入口

先按上文完成 Admin、Count 前端构建，再在仓库根目录使用 Composer 定义的发布命令。发布脚本会在必需的 `dist` 缺失时直接失败；不要手工执行 `zip`、复制 `dist` 或直接调用底层脚本：

```bash
# 用当前已构建的 dist 打包并校验候选包
composer release:build

# 校验本地或从 CI 下载的候选包
composer release:verify -- magick-toolbox.zip
```

构建产物固定写入仓库根目录：

- `magick-toolbox.zip`
- `magick-toolbox.zip.sha256`

ZIP 内部固定只有一个 `magick-toolbox/` 根目录，与 WordPress.org 插件 slug 一致。GitHub 仓库名和本地 checkout 目录仍为 `wp-magick-toolbox`，二者不要混用。

`composer release:build` 是本地构建发布包的唯一入口；`composer release:verify` 用于重新验证候选包。两者共同复用仓库内的发布边界，避免本地、CI 和人工发布各自维护一套排除规则。

### Push / PR 的 CI 制品

推送到 `main`、`develop`，或向 `main` 提交 Pull Request 时，GitHub Actions 会运行 PHP 语法检查、PHPUnit、PHPStan、前端类型与 lint、Vitest、前端构建和发布包验证。通过后上传名为 `magick-toolbox` 的 artifact，内容包括：

- `magick-toolbox.zip`
- `magick-toolbox.zip.sha256`

PR artifact 适合评审和候选验收；若合并后提交 SHA 发生变化，正式发布必须改用合并后 `main` 对应 CI run 的 artifact。

当前工作流只监听上述 branch push 和 pull request，**没有 `release` 事件触发器**。在 GitHub 创建 Release 不会自动构建、验证或上传插件 ZIP，也不会补救遗漏的 CI artifact。

### 手动发布

1. 更新版本、Stable tag 和 3.0.1 发布记录，提交后在本地运行 `composer release:build`。
2. 推送分支或提交 PR，等待目标提交的全部 CI 门禁通过。
3. 确认准备打标签的完整提交 SHA；如果经过 PR 合并，以合并后 `main` 的 SHA 为准。
4. 从该 SHA 对应的 GitHub Actions run 人工下载 `magick-toolbox` artifact，不使用其他 run、旧下载或本地重新打出的 ZIP 替代。
5. 在同一 SHA 的仓库检出中解压 artifact，并运行 `composer release:verify -- magick-toolbox.zip`；同时保留 CI 提供的 `.sha256` 文件。
6. 只有下载制品验证通过后，才在该**同一 SHA**上创建并推送 `v3.0.1` 标签。
7. 创建 GitHub Release，上传刚刚验证过的 `magick-toolbox.zip` 和 `magick-toolbox.zip.sha256`；不要在 Release 创建后重新构建或替换附件。

这条顺序保证测试的源码、打标签的源码和公开下载的二进制制品指向同一个提交。

### 发布包内容

发布包包含：
- 所有 PHP 文件（admin/, includes/, public/）
- 后台设置产物（vite/admin/dist/）
- 发文统计产物（vite/count/dist/）
- 插件主文件（magick-tool-box.php）
- 必要资源文件（css/, admin/partials/*/css/ 等）

发布包排除：
- 前端源码（vite/admin/src/、vite/count/src/）
- 前端工程文件（package.json、pnpm-lock.yaml、Vite/TypeScript/ESLint 配置等）
- 已退役的 `vite/public/`（目录本身必须不存在）
- 开发工具（tests/, docs/, ai/ 等）
- 依赖目录（node_modules/, vendor/）

## 目录结构

```
magick-toolbox/
├── admin/                    # 后台 PHP 代码
│   ├── class-magick-mixture-admin.php
│   ├── modules/              # 模块加载器
│   └── partials/             # 模块实现
├── includes/                 # 核心类
│   ├── class-magick-mixture.php
│   ├── class-magick-config-manager.php
│   └── ...
├── public/                   # 前台 PHP 代码
├── vite/                     # 单一前端工程（开发用）
│   ├── package.json          # 共享命令与依赖
│   ├── pnpm-lock.yaml        # 唯一前端锁文件
│   ├── admin/                # 管理面板源码、专用配置与 dist
│   └── count/                # 发文统计源码、专用配置与 dist
├── css/                      # 公共 CSS
└── magick-tool-box.php       # 插件主文件
```

## 注意事项

1. **不要提交 dist 目录**：`vite/admin/dist/` 与 `vite/count/dist/` 已在 `.gitignore` 中排除
2. **不要手工打包**：本地发布候选包只通过 `composer release:build` 生成，下载的 CI 候选包只通过 `composer release:verify` 复核
3. **版本号更新**：发布新版本时同步主文件 Header/常量、`readme.txt` Stable tag、README 和当前 changelog；WordPress `Tested up to` 单独按真实验证版本维护
4. **不要由 Release 反推制品**：当前 CI 不监听 GitHub Release；必须先取得同一提交 SHA 的通过制品，再创建标签和 Release
